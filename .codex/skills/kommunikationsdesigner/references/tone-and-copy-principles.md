# Tone and Copy Principles

## Default Tone

- Write in direct `du` form.
- Sound calm, competent, and useful.
- Prefer clear product language over marketing slogans.
- Avoid vague claims such as "revolutionary", "seamless", or "all-in-one" unless the UI and docs truly support them.

## Product Framing

- Describe HeuermannPlus as a tool for organizing groups and shared activities.
- Ground copy in the documented flows: registration, group creation, invitations, membership, and coordination.
- Treat future features carefully; if a capability is planned but not yet implemented, avoid presenting it as already available.

## CTA Rules

- Make the main action the shortest and clearest phrase on the surface.
- Use one dominant CTA per content block whenever possible.
- Keep secondary actions functional and less expressive than the primary CTA.
- Match the CTA to the next user step, not to an internal system concept.

## Microcopy Rules

- Labels: short, explicit, unambiguous.
- Helper text: explain why or what happens next; keep it brief.
- Empty states: explain the current situation and the next useful action.
- Status text: confirm what happened or what the user should expect next.
- Error-prone or sensitive flows: prefer precise language over friendly filler.

## Output Shape

When asked for copy, prefer this structure:

1. Final copy
2. One short alternative only if it solves a real tradeoff
3. Placement notes for UX or frontend when needed
4. Assumptions if product intent is incomplete

## Escalation Triggers

- Legal wording
- Authentication or verification wording with trust implications
- Security promises
- Claims about features not clearly present in docs or code
